# MCP와 OAuth 기초

이 문서는 "코드를 읽기 전에 꼭 알아야 하는 최소 개념"을 설명합니다. 목표는 엄밀한 표준 설명이 아니라, **이 저장소를 이해하는 데 필요한 수준까지 직관을 만드는 것**입니다.

## 1. MCP란 무엇인가

MCP는 쉽게 말해 **"클라이언트가 외부 도구를 일정한 형식으로 호출할 수 있게 해 주는 약속"**입니다.

초심자 관점에서 MCP를 아주 단순하게 이해하면 다음 정도면 충분합니다.

- 서버는 "내가 제공하는 도구 목록"을 공개한다.
- 클라이언트는 그 도구의 이름과 입력 형식을 보고 호출한다.
- 서버는 결과를 표준적인 형태로 반환한다.

이 프로젝트에서는 `greet`, `get_current_time`, `who_am_i`, `admin_ping`가 그런 도구입니다.

## 2. MCP에서 자주 나오는 구성요소

### MCP 서버

도구를 실제로 들고 있는 쪽입니다. 이 프로젝트에서는 [`fastmcp_client.py`](../fastmcp_client.py)가 서버를 띄웁니다.

### MCP 클라이언트

도구를 호출하는 쪽입니다. 여기서는 두 종류를 생각하면 됩니다.

- [`oauth/fastmcp_oauth.py`](../oauth/fastmcp_oauth.py) 같은 테스트용 Python 클라이언트
- ChatGPT 같은 실제 MCP 클라이언트

### Tool

클라이언트가 호출하는 함수 단위입니다. 보통 "원격으로 호출 가능한 함수"라고 생각하면 됩니다.

### Transport

클라이언트와 서버가 어떤 통신 방식으로 대화할지입니다. 이 프로젝트는 `http` transport로 서버를 실행합니다.

## 3. OAuth란 무엇인가

OAuth는 쉽게 말하면 **"비밀번호를 직접 넘기지 않고, 권한 토큰을 받아서 API를 쓰게 하는 방식"**입니다.

초심자는 아래 비유로 이해해도 충분합니다.

- 비밀번호 = 집 현관문 원키
- access token = 일정 시간만 유효한 출입증
- scope = 출입 가능한 구역

즉, 서버는 "이 사람이 누구인지"보다도 "이 요청이 어떤 권한(scope)을 가진 토큰을 들고 왔는지"를 더 중요하게 봅니다.

## 4. 이 프로젝트에서 OAuth가 필요한 이유

이 프로젝트의 도구는 단순하지만, 인증이 붙어 있기 때문에 다음을 할 수 있습니다.

- 인증된 사용자만 도구 호출
- scope에 따라 호출 가능한 도구 분리
- ChatGPT 같은 외부 클라이언트와 안전한 연결

예를 들어 `admin_ping`는 아무 토큰으로나 호출할 수 없고, `admin` scope가 있어야 합니다.

## 5. 꼭 알아야 할 OAuth 용어

### Access Token

API 호출에 직접 사용하는 토큰입니다. 이 프로젝트에서 클라이언트는 이 토큰을 들고 `/mcp`로 요청을 보냅니다.

### Refresh Token

Access token이 만료되었을 때 새 access token을 다시 발급받기 위한 토큰입니다. 보통 access token보다 더 오래 갑니다.

### Scope

"무슨 권한이 있는가"를 나타내는 문자열입니다.

이 프로젝트의 대표 scope:

- `mcp:use`
- `admin`

### Authorization Code

사용자가 로그인/승인을 끝냈을 때 잠깐 발급되는 일회용 코드입니다. 이 코드를 access token으로 교환합니다.

### Redirect URI

인증이 끝난 뒤 사용자를 어디로 돌려보낼지 나타내는 주소입니다. OAuth는 아무 주소로나 돌려보내면 위험하므로, 허용된 주소만 써야 합니다.

### PKCE

공개 클라이언트가 authorization code를 더 안전하게 교환하기 위해 쓰는 장치입니다. 이 프로젝트는 `code_challenge`를 보관합니다. 코드 전체를 보면 이 값 저장은 명시적으로 보이고, 이후 검증의 일부는 FastMCP/MCP의 상위 OAuth 구현 흐름에 기대는 것으로 보입니다. 이 부분은 **코드 구조를 보고 한 추론**입니다.

### Dynamic Client Registration

클라이언트가 미리 고정 등록되지 않아도, 실행 중에 자기 정보를 서버에 등록하는 방식입니다. ChatGPT 같은 클라이언트와 잘 맞습니다.

### OIDC

OAuth 위에 "사용자 신원 정보" 개념을 조금 더 얹은 표준입니다. Google, Auth0, Keycloak 등을 연결할 때 자주 등장합니다.

## 6. 이 프로젝트에서 누가 어떤 역할을 하나

OAuth 표준 용어로 보면 다음처럼 대응됩니다.

### Resource Server

보호된 자원, 즉 실제 도구를 들고 있는 서버입니다. 여기서는 MCP 서버가 이 역할을 합니다.

### Authorization Server

토큰을 발급하고 승인 과정을 처리하는 서버입니다. `custom` 모드에서는 [`oauth/chatgpt_oauth.py`](../oauth/chatgpt_oauth.py)가 이 역할까지 수행합니다.

### Client

토큰을 받아서 자원을 쓰려는 쪽입니다. ChatGPT나 [`oauth/fastmcp_oauth.py`](../oauth/fastmcp_oauth.py)가 여기 해당합니다.

### Resource Owner

권한을 승인하는 사용자입니다. 이 프로젝트에서는 개발용 JSON 사용자 목록에서 선택합니다.

## 7. Well-known 엔드포인트는 왜 필요한가

클라이언트는 "이 서버가 어떤 인증 방식을 쓰는지"를 알아야 합니다. 매번 사람에게 물어볼 수 없으니, 정해진 위치에 메타데이터를 둡니다.

이 프로젝트에서 중요한 URL:

- `/.well-known/oauth-authorization-server`
- `/.well-known/oauth-protected-resource/mcp`

이런 URL은 클라이언트에게 다음 정보를 알려주는 안내판 같은 역할을 합니다.

- authorization endpoint는 어디인가
- token endpoint는 어디인가
- 어떤 scope를 지원하는가
- 이 보호 자원의 주소는 무엇인가

## 8. 왜 ChatGPT 연결에는 공개 URL이 필요한가

여러분 노트북에서 `127.0.0.1:8000`으로 서버를 띄워도, ChatGPT는 그 주소를 볼 수 없습니다. ChatGPT는 인터넷을 통해 접근하므로, 다음이 필요합니다.

- 외부에서 접근 가능한 `https://...` 주소
- 그 주소와 정확히 맞는 OAuth 메타데이터
- 그 주소를 기준으로 생성되는 redirect/issuer/resource 설정

그래서 이 프로젝트에서는 `FASTMCP_BASE_URL`이 매우 중요합니다.

## 9. 이 저장소에서 인증이 없는 경우는 어떤 의미인가

`FASTMCP_OAUTH_PROVIDER=none`이면 인증 공급자를 붙이지 않습니다. 이 경우:

- OAuth 메타데이터가 필요 없다.
- 토큰 발급 흐름도 없다.
- `who_am_i`는 인증되지 않았다고 나온다.
- 도구 메타데이터에 scope 정보는 남아 있어도 실제 인증 provider가 없으므로 학습/디버깅용 상태가 된다.

즉, "MCP 서버 구조만 먼저 보고 싶을 때" 쓰는 모드에 가깝습니다.

## 10. 지금 단계에서 기억하면 되는 핵심

- MCP는 "도구 호출을 위한 공통 인터페이스"
- OAuth는 "토큰 기반 권한 부여"
- 이 프로젝트는 둘을 합쳐서 "인증된 MCP 서버"를 만든다
- 기본 모드에서는 커스텀 OAuth 공급자까지 직접 구현되어 있다
- 따라서 이 저장소는 서버 코드와 인증 코드가 함께 얽혀 있다
