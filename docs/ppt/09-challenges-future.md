## 1. 개발 과정에서의 주요 도전 (Challenges)

### 1.1 감시 백엔드 선택: inotifywait vs auditd

**도전 내용**

Linux 파일 시스템 이벤트를 수집하는 방식으로 `inotifywait`와 `auditd` 두 가지를 모두 검토했습니다. `inotifywait`는 설치가 간편하고 출력 형식이 단순하지만, 프로세스 정보가 없어 랜섬웨어 행위의 주체를 추적하기 어렵습니다. 반면 `auditd`는 `SYSCALL`, `PATH`, `CWD` 등 풍부한 프로세스 컨텍스트를 제공하지만, root 권한과 별도 규칙 설정이 필요하고 로그 파싱이 복잡합니다.

**적용한 해결책**

`src/collector/monitor-service.js`에서 `auto`, `auditd`, `inotify` 세 가지 모드를 지원하도록 구현했습니다. `auto` 모드에서는 먼저 `auditd`를 시도하고, 실패하면 `inotifywait`로 자동 폴백(fallback)합니다. 각 백엔드는 `InotifyMonitorBackend`와 `AuditdMonitorBackend` 클래스로 분리되어 있어, 향후 새로운 백엔드를 추가하기에도 용이한 구조입니다.

| 항목 | 상태 |
|------|------|
| 백엔드 자동 선택 및 폴백 | 구현 완료 |
| 백엔드별 이벤트 정규화 통합 | 구현 완료 |

### 1.2 이벤트 정규화: rename 쌍 맞추기와 불완전 이벤트 처리

**도전 내용**

`inotifywait`는 rename을 `MOVED_FROM`과 `MOVED_TO` 두 개의 독립 이벤트로 출력합니다. 이 둘을 쌍으로 맞추지 않으면 rename을 delete + create로 오인하거나, 반대로 delete 이벤트를 rename으로 오인할 수 있습니다. 또한 `auditd` 로그는 `SYSCALL`, `PATH`, `CWD` 등 여러 줄로 흩어져 있어 하나의 이벤트를 재구성해야 하며, 일부 레코드가 누락된 불완전 이벤트는 폐기해야 합니다.

**적용한 해결책**

`src/collector/monitor-event-parser.js`에서 `MonitorEventNormalizer`는 `MOVED_FROM`을 버퍼에 보관하고, `movePairWindowMs` 내에 같은 target 디렉터리로 `MOVED_TO`가 오면 `RENAME`으로 병합합니다. 시간이 초과하면 `MOVED_FROM`은 `DELETE`로 플러시됩니다. `AuditdEventNormalizer`는 `auditId` 기준으로 레코드를 그룹화하고, `SYSCALL`과 `PATH`가 모두 존재하며 `success=yes`인 경우에만 정규화합니다. `key` 필드로 감시 대상 필터링도 수행합니다.

| 항목 | 상태 |
|------|------|
| rename 이벤트 쌍 맞추기 | 구현 완료 |
| auditd 다중 레코드 병합 | 구현 완료 |
| 불완전 이벤트 필터링 | 구현 완료 |

### 1.3 탐지 규칙 임계값 조정: 탐지율과 오탐의 균형

**도전 내용**

랜섬웨어는 짧은 시간 내에 대량으로 파일을 수정하거나 이름을 바꾸므로, 이를 탐지하려면 낮은 임계값이 필요합니다. 하지만 정상적인 빌드 도구나 로그 로테이션도 짧은 시간 내에 많은 파일 이벤트를 발생시켜 오탐(false positive)을 유발할 수 있습니다. 단순 이벤트 개수 기반 탐지는 확장자별 위험도를 반영하지 못합니다.

**적용한 해결책**

`src/rules/rule-engine.js`에서 확장자 가중치(extension weight)와 이벤트 배율(event multiplier)을 결합한 `extension-weight-burst` 규칙을 구현했습니다. 예를 들어, 알려지지 않은 확장자로의 rename은 높은 가중치를 받고, 일반적인 소스 코드 수정은 상대적으로 낮은 가중치를 받습니다. 또한 `weightDecay` 메커니즘을 도입해 일정 시간이 지나면 가중치가 감소하도록 하여, 오래된 이벤트가 계속 누적되지 않도록 했습니다.

| 항목 | 상태 |
|------|------|
| 확장자 가중치 기반 탐지 | 구현 완료 |
| 가중치 감소(decay) 메커니즘 | 구현 완료 |
| 임계값 및 윈도우 세밀 조정 | 향후 개선 (docs/development-plan.md #3 참조) |

### 1.4 권한 기반 격리의 안전성: 범위 제한

**도전 내용**

탐지 시 감시 대상 디렉터리의 권한을 변경하여 격리하는 방식은 강력하지만, 잘못하면 시스템 전체에 영향을 미치거나 복구가 불가능해질 수 있습니다. 데모 환경이 아닌 실제 운영 환경에서의 안전성 확보가 중요했습니다.

**적용한 해결책**

`src/isolation/quarantine-service.js`에서 격리 범위를 `quarantineScope`로 제한합니다. 기본값은 `incident-target`으로, 탐지된 특정 디렉터리만 격리합니다. `all-watch-targets`로 확장할 수 있지만, 이는 명시적인 정책 설정을 통해서만 가능합니다. 또한 프로세스 종료(`killSuspectProcesses`)는 `demo-target` 경로 세그먼트를 포함하는 경로에만 허용되며, `/proc`를 통해 PID가 실제로 해당 경로를 점유하는지 재검증합니다. 격리 전 원래 권한을 메모리에 저장하고, `restore` API로 복원할 수 있도록 했습니다.

| 항목 | 상태 |
|------|------|
| 격리 범위 제한 (incident-target) | 구현 완료 |
| 프로세스 종료 안전 검증 | 구현 완료 |
| 원래 권한 저장 및 복구 | 구현 완료 |
| 격리 중복 처리 방지 | 구현 완료 (inProgressIds 추적) |

### 1.5 실시간 대시보드: WebSocket과 폴링 폴백

**도전 내용**

탐지 및 격리 결과를 웹 대시보드에 실시간으로 표시하려면 지속적인 연결이 필요합니다. 하지만 WebSocket이 항상 안정적으로 유지되지는 않으며, 네트워크 단절 시 대시보드가 멈춰 보일 수 있습니다.

**적용한 해결책**

`src/server/create-api-server.js`에서 hand-rolled WebSocket 서버를 구현하여, `FS_EVENT`, `RULE_MATCH`, `QUARANTINE_STARTED`, `QUARANTINE_COMPLETED`, `RESTORE_COMPLETED` 등 주요 이벤트를 브로드캐스트합니다. 대시보드(`public/app.js`)에서는 WebSocket 연결과 별개로 `setInterval(checkHealth, 5000)`을 통해 5초마다 `/api/health`를 호출합니다. `/api/snapshot`은 초기 로딩과 탐지·격리·복원·데모 상태 이벤트 이후에 호출하여 전체 상태를 다시 동기화합니다.

### 1.6 데모 시뮬레이션 안전성: 샌드박스와 되돌리기

**도전 내용**

랜섬웨어 유사 행위를 재현하는 데모는 실제 파일을 변경하므로, 시연 후 원래 상태로 되돌리지 못하면 문제가 됩니다. 또한 데모가 시스템 전체에 영향을 주지 않도록 격리된 환경에서 실행해야 합니다.

**적용한 해결책**

`src/simulator/demo.js`는 `demo-target` 디렉터리 내에서만 동작하도록 설계되었습니다. 파일 내용을 Base64 인코딩하고 확장자를 `.demo.locked`로 변경하며, 복원 시에는 `.demo.locked` 파일 안의 Base64 내용을 다시 디코딩해 원래 파일명으로 되돌립니다. `demo-worker.js`는 IPC를 통해 데모 진행 상태를 전달하고, `abort` 신호를 받으면 즉시 중단합니다. `ops/scripts/demo.sh`는 `start`와 `restore` 명령으로 데모 실행과 초기화를 분리했습니다.

| 항목 | 상태 |
|------|------|
| demo-target 샌드박스 제한 | 구현 완료 |
| Base64 변조 및 원상 복구 | 구현 완료 |
| 데모 중단(abort) 지원 | 구현 완료 |
| 데모 자동 초기화 | 향후 개선 (docs/development-plan.md #2 참조) |

## 3. 팀 회고 및 배운 점 (Reflections)

### 3.1 기술적 배움

- **Linux 파일 시스템 이벤트의 복잡성**: `inotify`와 `auditd`는 표면적으로 단순해 보이지만, 실제로는 이벤트 쌍 맞추기, 다중 레코드 병합, 프로세스 컨텍스트 추적 등 세밀한 처리가 필요했습니다. 이를 통해 커널 수준 이벤트가 사용자 공간으로 노출될 때의 정보 손실과 복원 문제를 직접 경험했습니다.
- **Node.js 스트림과 프로세스 관리**: `child_process.spawn`으로 실행한 `inotifywait`와 `tail -F`의 stdout을 `readline`으로 파싱하고, 프로세스 종료 시 trailing 이벤트를 플러시하는 과정에서 스트림 생명주기 관리의 중요성을 배웠습니다.
- **권한 기반 격리의 한계와 보완**: 단순히 `chmod`로 권한을 바꾸는 것 이상의 안전성 고려가 필요했습니다. 원래 권한 저장, 복구 실패 대응, 프로세스 종료 범위 제한 등 실제 운영 환경을 고려한 설계의 필요성을 느꼈습니다.

### 3.2 협업적 배움

- **모듈화와 인터페이스 설계**: `MonitorService`, `RuleEngine`, `IncidentStore`, `QuarantineService`를 `eventBus`로 연결하면서, 각 모듈의 책임을 명확히 분리하고 이벤트 계약(event contract)을 사전에 정의하는 것이 얼마나 중요한지 깨달았습니다. 이 덕분에 백엔드 교체나 규칙 개선이 다른 모듈에 미치는 영향을 최소화할 수 있었습니다.
- **문서화의 가치**: `docs/development-plan.md`와 `README.md`를 지속적으로 업데이트하면서, 팀원 간 지식 공유와 향후 유지보수를 위한 문서화 습관의 중요성을 체감했습니다.

### 3.3 아쉬운 점과 교훈

- **테스트 커버리지**: 핵심 기능에 대한 테스트는 구현했지만, 경계 조건(권한 없는 디렉터리, 비정상 종료, 네트워크 단절 등)에 대한 테스트가 부족했습니다. 향후에는 이러한 엣지 케이스를 더 적극적으로 다룰 계획입니다.
- **임계값 튜닝의 어려움**: 실제 다양한 워크로드에서의 오탐/미탐 데이터를 충분히 수집하지 못해, 탐지 규칙의 임계값이 이론적 수준에 머물러 있습니다. 운영 데이터 기반 튜닝의 필요성을 느꼈습니다.
- **데모와 실제의 간극**: 데모 시뮬레이션은 랜섬웨어 행위를 단순화하여 재현하므로, 알려진 랜섬웨어의 다양한 회피 기법(속도 조절, 선택적 암호화 등)을 모두 반영하지는 못했습니다. 이는 향후 탐지 규칙을 더욱 정교화해야 하는 동기가 되었습니다.
