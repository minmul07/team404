## 2. 프로젝트 배경

### 2.1 왜 랜섬웨어인가

랜섬웨어는 기업과 개인 모두에게 큰 피해를 주는 대표적인 사이버 위협입니다. Windows 환경에서는 백신과 EDR(Endpoint Detection and Response) 솔루션이 보편화되어 있지만, Linux 서버 환경에서는 상대적으로 실시간 행위 기반 탐지 도구가 부족합니다.

### 2.2 Linux 환경의 특수성

- 서버와 컨테이너 환경에서 Linux 사용 비중이 높음
- 기존 백신 중심 접근법은 Linux 파일 시스템 이벤트를 직접 활용하지 못함
- 짧은 시간 내 다수 파일 변경, 확장자 일괄 변경, 과도한 쓰기 작업 등은 랜섬웨어의 전형적인 공격 패턴

### 2.3 프로젝트 동기

본 프로젝트는 Linux 파일 시스템 이벤트를 직접 감시하여, 랜섬웨어와 유사한 비정상 행위를 조기에 탐지하고 자동으로 격리하는 1차 대응 체계를 구축하는 것을 목표로 합니다.

## 3. 프로젝트 목표 및 범위

### 3.1 핵심 목표

Linux 환경에서 특정 디렉터리 및 파일 시스템 이벤트를 감시하고, 랜섬웨어와 유사한 비정상 파일 행위가 발생했을 때 이를 탐지하여 자동으로 기록하고 격리하는 시스템을 구축합니다.

### 3.2 주요 기능

1. Linux 파일 시스템 이벤트 감시 (`src/collector/monitor-service.js`, `ops/config/monitor.sh`; 기본 `auto` 모드는 `auditd` 우선, 실패 시 `inotifywait` 폴백)
2. 짧은 시간 내 대량 수정/생성/이름 변경 탐지 (`src/rules/rule-engine.js`)
3. 랜섬웨어 유사 행위 발생 시 로그 저장 및 incident 생성 (`src/incidents/incident-store.js`)
4. 의심 행위 탐지 시 감시 대상 디렉터리 접근 권한 일시 제한 및 격리 (`src/isolation/quarantine-service.js`, `ops/scripts/quarantine.sh`)
5. 격리 결과를 웹 대시보드에 표시 및 로그 기록 (대시보드)
6. 시뮬레이션을 통한 탐지 데모 구성 (데모 모드)

### 3.3 판단 기준

- 일정 시간 내 다수 파일의 수정, 생성, 이름 변경 이벤트가 임계치를 초과할 경우 이상 행위로 판단
- 특정 확장자로의 일괄 변경, 짧은 시간 내 반복적 쓰기 작업, 다수 파일명 변경 등을 탐지 기준으로 사용

### 3.4 현재 개발 완료 상태

`docs/development-plan.md` 기준, 다음 항목이 구현 완료되었습니다.

- Linux 파일 시스템 이벤트 감시 및 `auditd`/`inotifywait` 백엔드 연동
- create / modify / delete / rename 이벤트 정규화
- threshold 기반 이상 행위 탐지 엔진
- incident 및 alert 상태 관리
- autoQuarantine 기반 자동 격리 및 restore 기반 권한 복구
- REST API 서버
- runtime 및 eventBus 기반 모듈 연결
- demo 모드 실행 구조
- 주요 기능 테스트 코드

남은 작업은 대시보드 마무리, 데모 시뮬레이션 보완, 탐지 규칙 개선, 문서화 업데이트 등입니다.

## 5. 차별점

### 5.1 단순 백신 흉내가 아닌 실제 파일 시스템 이벤트 활용

기존 백신 방식과 달리, 본 시스템은 Linux `auditd`와 `inotify` 기반 파일 시스템 이벤트를 직접 수집하고 분석합니다. 기본 `auto` 모드에서는 `auditd`를 먼저 시도하고, 사용할 수 없으면 `inotifywait` 기반 감시로 폴백합니다. 이를 통해 단순한 시그니처 매칭이 아닌, 행위 기반 탐지를 수행합니다.

### 5.2 감시에서 격리까지 이어지는 자동화 흐름

감시, 탐지, 기록, 격리까지 이어지는 랜섬웨어에 대한 1차적 대응 흐름을 하나의 파이프라인으로 구현했습니다. 탐지가 발생하면 즉시 incident가 생성되고, 설정에 따라 자동으로 격리가 실행됩니다.

### 5.3 웹 대시보드를 통한 실시간 가시성

텍스트 로그만으로는 파악하기 어려운 시스템 상태를 웹 대시보드를 통해 시각적으로 확인할 수 있습니다. health 상태, incident 목록, alert 목록, 격리 상태 등을 한눈에 볼 수 있습니다.

### 5.4 모듈화된 아키텍처

- `ops/`: 셸 스크립트 기반 저수준 감시 및 권한 제어
- `src/collector/`: 이벤트 수집 및 정규화
- `src/rules/`: 탐지 엔진
- `src/incidents/`: incident 및 alert 관리
- `src/isolation/`: 자동 격리 및 복구
- `src/server/`: REST API 제공

각 모듈은 eventBus를 통해 느슨하게 결합되어 있어, 확장과 유지보수가 용이합니다.

## 7. 시연 개요

발표 시 다음 흐름으로 8분 내외 데모를 진행할 예정입니다.

1. `npm run dev` 실행
2. demo-target 디렉터리 감시 시작
3. 랜섬웨어 유사 행위 발생 (데모 프로그램 실행)
4. `monitor.sh` 이벤트 감지
5. RuleEngine threshold 초과 탐지
6. incident 생성
7. autoQuarantine 실행
8. 파일 권한 잠금 수행
9. dashboard 및 API 상태 변경 확인
10. restore API 실행
11. 권한 복구 확인

> 참고: 악성코드를 제작하거나 실행하지 않으며, 짧은 시간 내 대량 파일 수정, 확장자 일괄 변경, 파일명 급변 등 랜섬웨어와 유사한 행동 패턴을 모의하여 탐지하는 방식으로 데모를 구성합니다.
